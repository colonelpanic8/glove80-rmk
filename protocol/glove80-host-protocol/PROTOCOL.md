# Glove80 host protocol v1.0

Wire format for the host ↔ keyboard control channel (Phase 2). One message
codec, transport-independent; a thin frame layer adapts it to USB raw HID
(32-byte reports) and a custom BLE GATT characteristic (ATT payloads, 20-byte
floor). Implementations: Rust (`protocol/glove80-host-protocol`, `no_std`) and
TypeScript (`ui/src/lib/host-protocol.ts`). Golden vectors:
`protocol/vectors/host-protocol-v1.json` — generated by the Rust test suite,
consumed by both.

## Conventions

- All multi-byte integers are **little-endian**.
- All sizes fixed; no varints, no compression.
- `u8` / `u16` / `u32` as usual.

## Versioning

- `PROTOCOL_VERSION_MAJOR = 1`, `PROTOCOL_VERSION_MINOR = 0`.
- Major bump: breaking change. Minor bump: additive (new commands, new
  effect kinds, new feature bits).
- Clients MUST send `GET_CAPABILITIES` first and MUST NOT assume any
  capacity, effect, or feature not advertised there.
- Unknown opcodes get a response with status `UNKNOWN_COMMAND`.

## Message layer

### Request

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | opcode (bit 7 always 0) |
| 1 | 1 | `request_id` — client-chosen, echoed in the response |
| 2 | 2 | `payload_len` (u16) |
| 4 | n | payload |

### Response

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | opcode of the request OR `0x80` (response flag) |
| 1 | 1 | `request_id` echoed |
| 2 | 1 | status |
| 3 | 2 | `payload_len` (u16) |
| 5 | n | payload |

- Every request gets exactly one response.
- A message is at most `MAX_MESSAGE_LEN = 1536` bytes including its header.
- Decoders MUST reject messages whose length ≠ header length + `payload_len`.

### Status codes

| value | name | meaning |
| --- | --- | --- |
| 0x00 | `OK` | applied fully |
| 0x01 | `UNKNOWN_COMMAND` | opcode not understood |
| 0x02 | `MALFORMED` | payload failed to parse |
| 0x03 | `OUT_OF_RANGE` | key index / value outside advertised capacity |
| 0x04 | `CAPACITY_EXCEEDED` | batch larger than `max_cells_per_op` or overlay full |
| 0x05 | `PARTIAL_APPLY` | applied on central; peripheral portion pending (see overlay ack) |
| 0x06 | `BUSY` | try again |
| 0x07 | `UNKNOWN_TOGGLE` | toggle id not configured |
| 0x08 | `BAD_MAGIC` | bootloader entry without the magic constant |
| 0x09 | `UNSUPPORTED_VERSION` | client major version not supported |

- Status `OK` and `PARTIAL_APPLY` carry the command-specific payload below;
  every other status carries an **empty** payload.
- `PARTIAL_APPLY` is only valid on the four overlay-write commands.

## Commands

| opcode | name | request payload | ok-response payload |
| --- | --- | --- | --- |
| 0x01 | `GET_CAPABILITIES` | client version | capabilities |
| 0x02 | `PING` | 0–64 opaque bytes | same bytes echoed |
| 0x10 | `SET_CELLS` | ttl + cell batch | overlay ack |
| 0x11 | `UNSET_CELLS` | key list | overlay ack |
| 0x12 | `CLEAR_OVERLAY` | empty | overlay ack |
| 0x13 | `READ_OVERLAY` | empty | overlay state |
| 0x14 | `REPLACE_OVERLAY` | ttl + cell batch | overlay ack |
| 0x20 | `GET_BRIGHTNESS` | empty | brightness |
| 0x21 | `SET_BRIGHTNESS` | level (u8) | brightness (as applied) |
| 0x30 | `GET_TOGGLE` | id (u8) | toggle state |
| 0x31 | `SET_TOGGLE` | id + state | toggle state (as applied) |
| 0x7F | `ENTER_BOOTLOADER` | magic + target | empty |

### GET_CAPABILITIES (0x01)

Request payload (2 bytes): `client_major u8`, `client_minor u8`.

Response payload (16 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `protocol_major` |
| 1 | 1 | `protocol_minor` |
| 2 | 1 | `led_count_left` (40 on Glove80) |
| 3 | 1 | `led_count_right` (40) |
| 4 | 1 | `layer_capacity` (8) |
| 5 | 1 | `max_cells_per_op` — max cells in one SET/REPLACE, max keys in one UNSET |
| 6 | 2 | `effect_mask` — bit n set ⇔ effect kind n supported |
| 8 | 2 | `overlay_cell_capacity` — max simultaneously set host-overlay cells |
| 10 | 2 | `max_message_len` device accepts |
| 12 | 4 | `feature_bits` |

Feature bits:

| bit | feature |
| --- | --- |
| 0 | per-write TTL |
| 1 | toggles |
| 2 | programmatic bootloader entry |
| 3 | atomic replace |
| 4 | overlay read-back |
| 5 | partial-apply reporting |

Key index space: `0 .. led_count_left + led_count_right - 1`. Left half is
`0..led_count_left`, right half follows.

### PING (0x02)

Request payload: 0–64 opaque bytes. Response payload: the same bytes.

### Effect record (10 bytes, used by cell payloads)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `kind` — 0 solid, 1 blink, 2 breathe |
| 1 | 1 | `r` |
| 2 | 1 | `g` |
| 3 | 1 | `b` |
| 4 | 2 | `period_ms` (u16) |
| 6 | 2 | `phase_ms` (u16) |
| 8 | 1 | `duty_percent` (0–100) |
| 9 | 1 | reserved — encode 0, decoders ignore |

- Solid ignores `period_ms`/`phase_ms`/`duty_percent`; breathe ignores
  `duty_percent`. Encoders SHOULD write 0 for ignored fields; decoders MUST
  preserve whatever is present (round-trip exact).
- New effect kinds reuse this record unchanged and are gated by
  `effect_mask` — existing records never change meaning.

### SET_CELLS (0x10) / REPLACE_OVERLAY (0x14)

Request payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 4 | `ttl_ms` (u32) — applies to every cell in this write; 0 = no TTL |
| 4 | 1 | `cell_count` |
| 5 | 11·n | cells: `key u8` + effect record (10 bytes) |

- `SET_CELLS` merges into the overlay; `REPLACE_OVERLAY` atomically replaces
  the entire overlay (unlisted keys become transparent). `REPLACE_OVERLAY`
  with `cell_count = 0` is equivalent to `CLEAR_OVERLAY`.
- On TTL expiry the cell reverts to transparent (firmware-enforced).

### UNSET_CELLS (0x11)

Request payload: `key_count u8`, then `key_count` × `key u8`.

### CLEAR_OVERLAY (0x12) / READ_OVERLAY (0x13)

Empty request payloads.

### Overlay ack (response payload for 0x10/0x11/0x12/0x14)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `pending_count` |
| 1 | n | pending key indices (u8 each) |

- `pending_count = 0` with status `OK`: fully applied on both halves.
- Status `PARTIAL_APPLY`: listed keys were accepted on the central but not
  yet applied to the (offline) peripheral. For `CLEAR_OVERLAY`, a pending
  clear on the offline half is reported as `PARTIAL_APPLY` with
  `pending_count = 0`.

### Overlay state (response payload for 0x13)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `cell_count` |
| 1 | 15·n | entries: `key u8` + effect record (10) + `remaining_ttl_ms u32` |

- `remaining_ttl_ms = 0` means no TTL on that cell.

### GET_BRIGHTNESS (0x20) / SET_BRIGHTNESS (0x21)

- `SET_BRIGHTNESS` request payload: `level u8` (0–255 global scalar; the
  driver's current ceiling still clamps on top).
- Both respond with `level u8` — the value now in effect.

### GET_TOGGLE (0x30) / SET_TOGGLE (0x31)

- `GET_TOGGLE` request payload: `id u8`.
- `SET_TOGGLE` request payload: `id u8`, `state u8` (0 off, 1 on).
- Both respond with `id u8`, `state u8` — the state now in effect.

### ENTER_BOOTLOADER (0x7F)

Request payload (5 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 4 | magic (u32) = `0xB00710AD` |
| 4 | 1 | target: 0 central, 1 peripheral |

- Wrong magic → status `BAD_MAGIC`, nothing happens.
- The empty-payload OK response may never arrive (device resets); clients
  MUST tolerate that.

## Frame layer (per-transport segmentation)

Every transport chunk (one HID report / one ATT write or notification):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | control: bits 0–6 = sequence (0-based, per message), bit 7 = FINAL |
| 1 | 1 | `chunk_payload_len` (bytes of message data in this chunk) |
| 2 | n | message bytes |
| 2+n | — | padding (zeros) up to the transport's fixed chunk size, if any |

- Sequence restarts at 0 for each message; max 128 frames per message.
- One message in flight per direction per transport; interleaving is not
  allowed. A frame with sequence 0 always starts a new message (an
  incomplete previous message is dropped).
- A sequence gap or a chunk shorter than `2 + chunk_payload_len` aborts
  reassembly with an error.
- USB raw HID: chunk size 32, reports zero-padded (payload ≤ 30/report).
- BLE GATT: chunk size = negotiated ATT payload, assume ≥ 20 (payload ≥ 18
  per write). Frames need not be padded.
- The message header's `payload_len` cross-checks reassembly: total
  reassembled length must equal header length + `payload_len`.

## Transports

How the frame layer maps onto the physical transports the Glove80 RMK
firmware exposes (Phase 2). Both transports terminate on the **central (left)
half**; the peripheral has no host-facing transport. One request/response
exchange is in flight per transport at a time (enforced device-side: the next
request frame is not consumed until the previous response is fully framed
out).

### USB raw HID

- A dedicated vendor raw-HID interface on the keyboard's composite USB
  device (VID `0x16C0`, PID `0x27DB`), separate from the keyboard, composite
  and Vial interfaces.
- Match it by HID usage page **`0xFF88`**, usage **`0x01`** (Vial's raw-HID
  interface is usage page `0xFF60`/usage `0x61` — do not confuse them; the
  opcode spaces collide).
- Reports: 32-byte IN and 32-byte OUT, no report IDs. Every report is one
  frame-layer chunk, zero-padded to 32 bytes (≤ 30 payload bytes/report).

### BLE GATT

- A custom primary service (deliberately **not** a HID service, so Web
  Bluetooth can reach it; it is not in the advertising payload — Web
  Bluetooth clients must list it in `optionalServices`):

| role | UUID |
| --- | --- |
| service | `fc550001-f8e0-459f-b421-c254fc42b138` |
| request characteristic (host → keyboard) | `fc550002-f8e0-459f-b421-c254fc42b138` |
| response characteristic (keyboard → host) | `fc550003-f8e0-459f-b421-c254fc42b138` |

- Requests: **write-without-response** to the request characteristic, one
  frame-layer chunk per write, unpadded. Chunk size ≤
  `min(negotiated ATT payload, 257)` (257 = 2-byte frame header + the
  255-byte max chunk payload).
- Responses: **notifications** on the response characteristic (subscribe via
  CCCD first), one chunk per notification, unpadded, sized to
  `min(negotiated ATT payload, 257)` as observed when the request arrived.
- Like the rest of the GATT database, the service requires an encrypted
  (bonded) connection.

### Phase 2 device status

- Key indices `0..40` (left half) apply immediately on the central. Indices
  `40..80` (right half) are accepted and acknowledged as pending via
  `PARTIAL_APPLY`, but until split forwarding lands (Phase 3) the pending
  cells are **dropped**: they never render, do not appear in `READ_OVERLAY`,
  and `CLEAR_OVERLAY` answers plain `OK` (the right half can hold no host
  cells yet, so a clear is trivially complete).
- `ENTER_BOOTLOADER` target 1 (peripheral) is not yet reachable and answers
  `OUT_OF_RANGE` (the protocol has no dedicated "unsupported" status);
  target 0 (central) works via the Adafruit bootloader.
