# Glove80 host protocol v1.4

Wire format for the host ‚Üî keyboard control channel (Phase 2). One message
codec, transport-independent; a thin frame layer adapts it to USB raw HID
(32-byte reports) and a custom BLE GATT characteristic (ATT payloads, 20-byte
floor). Implementations: Rust (`crates/glove80-host-protocol`, `no_std`) and
TypeScript (`ui/src/lib/host-protocol.ts`). Golden vectors, generated by the
Rust test suite and consumed by both codecs:
`crates/glove80-host-protocol/vectors/host-protocol-v1.json` (v1.0, frozen ‚Äî bytes never change),
`crates/glove80-host-protocol/vectors/host-protocol-v1.1.json` (the v1.1 additions, frozen),
`crates/glove80-host-protocol/vectors/host-protocol-v1.2.json` (the v1.2 additions, frozen) and
`crates/glove80-host-protocol/vectors/host-protocol-v1.3.json` (the v1.3 additions, frozen) and
`crates/glove80-host-protocol/vectors/host-protocol-v1.4.json` (conditional-lighting gates).

## Conventions

- All multi-byte integers are **little-endian**.
- All sizes fixed; no varints, no compression.
- `u8` / `u16` / `u32` as usual.

## Versioning

- `PROTOCOL_VERSION_MAJOR = 1`, `PROTOCOL_VERSION_MINOR = 4`.
- v1.1 is purely additive over v1.0: the CONFIG_* commands (0x40‚Äì0x44), five
  config statuses (0x0A‚Äì0x0E), feature bit 6, the capability extension, and
  the config blob format. Every v1.0 message is byte-identical under v1.1.
- v1.2 is purely additive over v1.1: the KEYMAP_* commands (0x50‚Äì0x51),
  feature bit 7, and the keymap capability extension. No new statuses. Every
  v1.0/v1.1 message is byte-identical under v1.2.
- v1.3 is purely additive over v1.2: the GET_VERSION command (0x03) and
  feature bit 8. No new statuses, no capability extension. Every
  v1.0/v1.1/v1.2 message is byte-identical under v1.3.
- v1.4 is a backward-compatible extension of the v1 config blob: feature bit
  9 advertises per-record gates encoded in the record header's former
  reserved `u16`. Ungated records retain the all-zero bytes, so every frozen
  v1.0‚Äìv1.3 message and every ungated v1.1 blob remains byte-identical.
- Major bump: breaking change. Minor bump: additive (new commands, new
  effect kinds, new feature bits).
- Clients MUST send `GET_CAPABILITIES` first and MUST NOT assume any
  capacity, effect, or feature not advertised there.
- Unknown opcodes get a response with status `UNKNOWN_COMMAND`.

## Message layer

### Request

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | opcode (bit 7 always 0) |
| 1 | 1 | `request_id` ‚Äî client-chosen, echoed in the response |
| 2 | 2 | `payload_len` (u16) |
| 4 | n | payload |

### Response

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | opcode of the request OR `0x80` (response flag) |
| 1 | 1 | `request_id` echoed |
| 2 | 1 | status |
| 3 | 2 | `payload_len` (u16) |
| 5 | n | payload |

- Every request gets exactly one response.
- A message is at most `MAX_MESSAGE_LEN = 1536` bytes including its header.
- Decoders MUST reject messages whose length ‚âÝ header length + `payload_len`.

### Status codes

| value | name | meaning |
| --- | --- | --- |
| 0x00 | `OK` | applied fully |
| 0x01 | `UNKNOWN_COMMAND` | opcode not understood |
| 0x02 | `MALFORMED` | payload failed to parse |
| 0x03 | `OUT_OF_RANGE` | key index / value outside advertised capacity |
| 0x04 | `CAPACITY_EXCEEDED` | batch larger than `max_cells_per_op` or overlay full |
| 0x05 | `PARTIAL_APPLY` | applied on central; peripheral portion pending (see overlay ack) |
| 0x06 | `BUSY` | try again |
| 0x07 | `UNKNOWN_TOGGLE` | toggle id not configured |
| 0x08 | `BAD_MAGIC` | bootloader entry without the magic constant |
| 0x09 | `UNSUPPORTED_VERSION` | client major version not supported |
| 0x0A | `NO_SESSION` | CONFIG_DATA/COMMIT without an open transfer session (v1.1) |
| 0x0B | `BAD_OFFSET` | CONFIG_DATA offset not contiguous, or past the announced total (v1.1) |
| 0x0C | `CONFIG_INCOMPLETE` | CONFIG_COMMIT before all announced bytes arrived (v1.1) |
| 0x0D | `CRC_MISMATCH` | assembled blob fails the announced or header CRC (v1.1) |
| 0x0E | `INVALID_CONFIG` | blob failed validation; nothing changed (v1.1) |

- Status `OK` and `PARTIAL_APPLY` carry the command-specific payload below;
  every other status carries an **empty** payload.
- `PARTIAL_APPLY` is only valid on the four overlay-write commands.
- Statuses 0x0A‚Äì0x0E only ever answer the CONFIG_* commands, which a v1.0
  client never sends ‚Äî old decoders cannot encounter them.

## Commands

| opcode | name | request payload | ok-response payload |
| --- | --- | --- | --- |
| 0x01 | `GET_CAPABILITIES` | client version | capabilities |
| 0x02 | `PING` | 0‚Äì64 opaque bytes | same bytes echoed |
| 0x03 | `GET_VERSION` | empty | both halves' build identity (v1.3) |
| 0x10 | `SET_CELLS` | ttl + cell batch | overlay ack |
| 0x11 | `UNSET_CELLS` | key list | overlay ack |
| 0x12 | `CLEAR_OVERLAY` | empty | overlay ack |
| 0x13 | `READ_OVERLAY` | empty | overlay state |
| 0x14 | `REPLACE_OVERLAY` | ttl + cell batch | overlay ack |
| 0x20 | `GET_BRIGHTNESS` | empty | brightness |
| 0x21 | `SET_BRIGHTNESS` | level (u8) | brightness (as applied) |
| 0x30 | `GET_TOGGLE` | id (u8) | toggle state |
| 0x31 | `SET_TOGGLE` | id + state | toggle state (as applied) |
| 0x40 | `CONFIG_BEGIN` | total_len + blob crc32 | empty (v1.1) |
| 0x41 | `CONFIG_DATA` | offset + bytes | empty (v1.1) |
| 0x42 | `CONFIG_COMMIT` | empty | empty (v1.1) |
| 0x43 | `CONFIG_ABORT` | empty | empty (v1.1) |
| 0x44 | `CONFIG_READ` | offset + max_len | total_len + bytes (v1.1) |
| 0x50 | `KEYMAP_READ` | layer + start_key + max_count | keycode batch (v1.2) |
| 0x51 | `KEYMAP_WRITE` | keymap entry batch | per-entry read-back (v1.2) |
| 0x7F | `ENTER_BOOTLOADER` | magic + target | empty |

### GET_CAPABILITIES (0x01)

Request payload (2 bytes): `client_major u8`, `client_minor u8`.

Response payload (16 bytes; +4 with the v1.1 extension, +4 more with the
v1.2 extension):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `protocol_major` |
| 1 | 1 | `protocol_minor` |
| 2 | 1 | `led_count_left` (40 on Glove80) |
| 3 | 1 | `led_count_right` (40) |
| 4 | 1 | `layer_capacity` (8) |
| 5 | 1 | `max_cells_per_op` ‚Äî max cells in one SET/REPLACE, max keys in one UNSET |
| 6 | 2 | `effect_mask` ‚Äî bit n set ‚áî effect kind n supported |
| 8 | 2 | `overlay_cell_capacity` ‚Äî max simultaneously set host-overlay cells |
| 10 | 2 | `max_message_len` device accepts |
| 12 | 4 | `feature_bits` |
| 16 | 4 | `max_config_blob_len` (u32) ‚Äî present **iff** feature bit 6 is set |
| +0 | 1 | `keymap_rows` ‚Äî present **iff** feature bit 7 is set (v1.2) |
| +1 | 1 | `keymap_cols` (v1.2) |
| +2 | 1 | `max_keymap_entries_per_op` ‚Äî max entries in one KEYMAP_READ/WRITE (v1.2) |
| +3 | 1 | reserved ‚Äî encode 0, decoders ignore (v1.2) |

Feature bits:

| bit | feature |
| --- | --- |
| 0 | per-write TTL |
| 1 | toggles |
| 2 | programmatic bootloader entry |
| 3 | atomic replace |
| 4 | overlay read-back |
| 5 | partial-apply reporting |
| 6 | persistent config (CONFIG_* commands, v1.1) |
| 7 | keymap editing (KEYMAP_* commands, v1.2) |
| 8 | build-identity reporting (GET_VERSION, v1.3; no capability extension) |
| 9 | conditional-lighting gates in config records (v1.4; no capability extension) |

- The payload is extended strictly by appending, gated by feature bits:
  `max_config_blob_len` (the largest blob CONFIG_BEGIN accepts) is on the
  wire exactly when bit 6 is set; the 4-byte keymap extension follows it
  exactly when bit 7 is set (extensions appear in feature-bit order ‚Äî a
  device with bit 7 but not bit 6 appends the keymap extension directly at
  offset 16). Decoders treat an absent extension as 0.

Key index space: `0 .. led_count_left + led_count_right - 1`. Left half is
`0..led_count_left`, right half follows.

### PING (0x02)

Request payload: 0‚Äì64 opaque bytes. Response payload: the same bytes.

### GET_VERSION (0x03)

Firmware build identity of **both** halves in one exchange (v1.3, gated by
feature bit 8). Empty request payload.

OK response payload (33 bytes): a 16-byte half entry for the central, the
same for the peripheral, then one flag byte.

Half entry (16 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `present` ‚Äî 0 or 1, see below |
| 1 | 1 | `fw_major` |
| 2 | 1 | `fw_minor` |
| 3 | 1 | `fw_patch` |
| 4 | 8 | `git_hash` ‚Äî ASCII short commit hash, zero-padded on the right |
| 12 | 1 | `dirty` ‚Äî 1 ‚áî built from a tree with uncommitted changes |
| 13 | 3 | reserved ‚Äî encode 0, decoders ignore |

Full payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 16 | central half entry |
| 16 | 16 | peripheral half entry |
| 32 | 1 | `halves_mismatch` ‚Äî 0 or 1, computed by the firmware |

- `fw_major.minor.patch` is the firmware crate's semver; `git_hash` is
  `git rev-parse --short=8 HEAD` of the build tree. A build without git
  available reports the literal ASCII `"unknown0"` with `dirty = 0`.
- `present`: the central is always present in its own response. The
  peripheral entry reports the version it announced over the split link;
  while the link is down `present = 0` but the last-known version fields
  are retained (all-zero fields ‚áî the peripheral was never seen since the
  central booted).
- `halves_mismatch` is 1 exactly when both entries are present and their
  `git_hash` or firmware semver differ. Hosts SHOULD surface it prominently
  ‚Äî mismatched halves are a flash-one-half-and-forgot-the-other state.
- `present`, `dirty` and `halves_mismatch` MUST be 0 or 1; decoders reject
  other values.

### Effect record (10 bytes, used by cell payloads)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `kind` ‚Äî 0 solid, 1 blink, 2 breathe |
| 1 | 1 | `r` |
| 2 | 1 | `g` |
| 3 | 1 | `b` |
| 4 | 2 | `period_ms` (u16) |
| 6 | 2 | `phase_ms` (u16) |
| 8 | 1 | `duty_percent` (0‚Äì100) |
| 9 | 1 | reserved ‚Äî encode 0, decoders ignore |

- Solid ignores `period_ms`/`phase_ms`/`duty_percent`; breathe ignores
  `duty_percent`. Encoders SHOULD write 0 for ignored fields; decoders MUST
  preserve whatever is present (round-trip exact).
- New effect kinds reuse this record unchanged and are gated by
  `effect_mask` ‚Äî existing records never change meaning.

### SET_CELLS (0x10) / REPLACE_OVERLAY (0x14)

Request payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 4 | `ttl_ms` (u32) ‚Äî applies to every cell in this write; 0 = no TTL |
| 4 | 1 | `cell_count` |
| 5 | 11¬∑n | cells: `key u8` + effect record (10 bytes) |

- `SET_CELLS` merges into the overlay; `REPLACE_OVERLAY` atomically replaces
  the entire overlay (unlisted keys become transparent). `REPLACE_OVERLAY`
  with `cell_count = 0` is equivalent to `CLEAR_OVERLAY`.
- On TTL expiry the cell reverts to transparent (firmware-enforced).

### UNSET_CELLS (0x11)

Request payload: `key_count u8`, then `key_count` √ó `key u8`.

### CLEAR_OVERLAY (0x12) / READ_OVERLAY (0x13)

Empty request payloads.

### Overlay ack (response payload for 0x10/0x11/0x12/0x14)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `pending_count` |
| 1 | n | pending key indices (u8 each) |

- `pending_count = 0` with status `OK`: fully applied on both halves.
- Status `PARTIAL_APPLY`: listed keys were accepted on the central but not
  yet applied to the (offline) peripheral. For `CLEAR_OVERLAY`, a pending
  clear on the offline half is reported as `PARTIAL_APPLY` with
  `pending_count = 0`.

### Overlay state (response payload for 0x13)

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `cell_count` |
| 1 | 15¬∑n | entries: `key u8` + effect record (10) + `remaining_ttl_ms u32` |

- `remaining_ttl_ms = 0` means no TTL on that cell.

### GET_BRIGHTNESS (0x20) / SET_BRIGHTNESS (0x21)

- `SET_BRIGHTNESS` request payload: `level u8` (0‚Äì255 global scalar; the
  driver's current ceiling still clamps on top).
- Both respond with `level u8` ‚Äî the value now in effect.

### GET_TOGGLE (0x30) / SET_TOGGLE (0x31)

- `GET_TOGGLE` request payload: `id u8`.
- `SET_TOGGLE` request payload: `id u8`, `state u8` (0 off, 1 on).
- Both respond with `id u8`, `state u8` ‚Äî the state now in effect.

### ENTER_BOOTLOADER (0x7F)

Request payload (5 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 4 | magic (u32) = `0xB00710AD` |
| 4 | 1 | target: 0 central, 1 peripheral |

- Wrong magic ‚Üí status `BAD_MAGIC`, nothing happens.
- The empty-payload OK response may never arrive (device resets); clients
  MUST tolerate that.

## Persistent configuration (v1.1)

Persistent lighting configuration (base / layer / toggle records plus toggle
persistence) travels as one **config blob** ‚Äî the unit of persistence and
transfer. Hosts apply it transactionally (CONFIG_BEGIN ‚Üí CONFIG_DATA‚Ä¶ ‚Üí
CONFIG_COMMIT): the device activates and persists either the complete new
blob or keeps the old one, never a hybrid. CONFIG_READ exports the active
blob **byte-stable** ‚Äî exactly the bytes that were committed. Gated by
feature bit 6; blob size bounded by the advertised `max_config_blob_len`.

### Config blob format

Versioned separately from the protocol (`config_version = 1`). All
little-endian. Header (16 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 4 | magic (u32) = `0x4C303847` ("G80L") |
| 4 | 2 | version (u16) = 1 |
| 6 | 2 | reserved ‚Äî encode 0 |
| 8 | 4 | `body_len` (u32) |
| 12 | 4 | `body_crc32` (u32) ‚Äî CRC-32/ISO-HDLC over the body bytes |

Body, starting at offset 16 (12-byte prefix, then the records):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `record_count` (‚â§ 16) |
| 1 | 4 | `toggle_persist_mask` (u32) ‚Äî bit n: toggle n's runtime state persists across reboots (opt-in) |
| 5 | 4 | `toggle_initial_state` (u32) ‚Äî bit n: toggle n's boot state (for non-persisted toggles) |
| 9 | 3 | reserved ‚Äî encode 0 |
| 12 | ‚Ä¶ | `record_count` records, back to back |

Each record:

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `activation` ‚Äî 0 Always, 1 LayerActive, 2 Toggle |
| 1 | 1 | `activation_arg` ‚Äî layer (< 8) / toggle id (< 32); 0 for Always |
| 2 | 1 | `gate_kind` ‚Äî 0 None, 1 LayerActive, 2 Toggle, 3 UsbConnected, 4 Charging, 5 SplitLinkUp (v1.4; formerly reserved) |
| 3 | 1 | `gate_arg` ‚Äî layer (< 8) / toggle id (< 32); 0 for None and state conditions (v1.4; formerly reserved) |
| 4 | 1 | `cell_count` (‚â§ 40) |
| 5 | 11¬∑n | cells: `key u8` (< 80) + effect record (10 bytes, unchanged from v1) |

- Host-overlay and status records are runtime state and **not persistable**;
  only activations 0‚Äì2 exist in a blob.
- A gate is an optional second predicate ANDed with the record activation.
  `gate_kind = 0, gate_arg = 0` means no gate and is exactly the encoding all
  pre-v1.4 blobs already used for the reserved `u16`. Kinds 1 and 2 use the
  same layer/toggle ranges as activations; kinds 3‚Äì5 require `gate_arg = 0`.
- Hosts MUST only send nonzero gate kinds when capability feature bit 9 is
  advertised. Firmware that does not recognize a gate kind rejects the blob
  at CONFIG_COMMIT with `INVALID_CONFIG`; it does not partially apply it.
- Record order in the blob = composition order within each activation class.
- Records are sparse maps: an unlisted key is transparent. A key MUST NOT
  appear twice in one record (validation rejects duplicates).
- CRC-32/ISO-HDLC is the zlib/PNG CRC (reflected poly `0xEDB88320`, init and
  xor-out `0xFFFFFFFF`); `crc32("123456789") = 0xCBF43926`.
- Maximum blob: 16 + 12 + 16¬∑(5 + 40¬∑11) = **7148 bytes**
  (`MAX_CONFIG_BLOB_LEN`); devices may advertise a smaller
  `max_config_blob_len`.
- Validation (all enforced by both codecs and by firmware before commit):
  magic, version, `body_len` = actual body length, body CRC, `record_count`
  ‚â§ 16, `cell_count` ‚â§ 40, activation kind known, layer arg < 8, toggle arg
  < 32, gate kind known, gate layer arg < 8, gate toggle arg < 32,
  firmware-state gate args = 0, key < 80, no duplicate key per record,
  effect kind known, no trailing bytes.

### Transfer session (apply direction)

- `CONFIG_BEGIN` (0x40) request payload (8 bytes): `total_len u32`,
  `blob_crc32 u32` ‚Äî length and CRC-32/ISO-HDLC of the **entire blob**
  (header + body). Opens the one transfer session; a new BEGIN silently
  replaces any open session. `total_len` beyond `max_config_blob_len` ‚Üí
  `CAPACITY_EXCEEDED`. OK response: empty.
- `CONFIG_DATA` (0x41) request payload: `offset u32` + the chunk bytes (the
  rest of the payload, ‚â§ `MAX_CONFIG_DATA_PER_MESSAGE` = 1024). Chunks are
  strictly sequential: `offset` MUST equal the byte count received so far
  (no gaps, no overlap, no rewind), and `offset + len` MUST NOT exceed
  `total_len`. OK response: empty.
- `CONFIG_COMMIT` (0x42), empty payload: with all `total_len` bytes present,
  the device checks the announced blob CRC, then fully validates the blob
  (list above), then **atomically activates and persists** it. OK means the
  new config is active and durable. Any failure leaves the previous config
  active and untouched.
- `CONFIG_ABORT` (0x43), empty payload: discards any open session.
  Idempotent ‚Äî answers OK even when no session is open.

Session state machine:

```text
idle ‚îÄ‚îÄCONFIG_BEGIN‚îÄ‚îÄ‚ñ∫ transferring ‚îÄ‚îÄCONFIG_COMMIT(valid)‚îÄ‚îÄ‚ñ∫ idle (new config active+persisted)
  ‚ñ≤                        ‚îÇ CONFIG_DATA (contiguous, loops)
  ‚îÇ                        ‚îú‚îÄCONFIG_COMMIT(failure)‚îÄ‚ñ∫ idle (old config kept)
  ‚îî‚îÄ‚îÄ‚îÄ‚îÄCONFIG_ABORT‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò (also: any CONFIG_DATA error, or a new BEGIN replacing the session)
```

- Exactly one session exists; it lives on the central across both transports
  (a BEGIN on one transport replaces a session opened on the other).
- Error conditions:
  - `CONFIG_DATA`/`CONFIG_COMMIT` with no session ‚Üí `NO_SESSION`.
  - Non-contiguous or overflowing `CONFIG_DATA` ‚Üí `BAD_OFFSET`, and the
    session is aborted (restart with BEGIN).
  - `CONFIG_COMMIT` with `received < total_len` ‚Üí `CONFIG_INCOMPLETE`.
  - Assembled bytes fail the announced `blob_crc32` (or the header's
    `body_crc32`) ‚Üí `CRC_MISMATCH`.
  - Structural/range validation fails ‚Üí `INVALID_CONFIG`.
  - Every `CONFIG_COMMIT`, success or failure, ends the session.
- A reboot or power loss mid-session discards the partial transfer; the
  previous persisted config still boots (a malformed or interrupted write
  can never strand the keyboard).

### CONFIG_READ (0x44) ‚Äî export

- Request payload (6 bytes): `offset u32`, `max_len u16`.
- OK response payload: `total_len u32` (full length of the active blob) +
  the blob bytes at `offset`, at most `min(max_len, 1024)` of them (possibly
  fewer; empty exactly at `offset = total_len`).
- `offset > total_len` ‚Üí `OUT_OF_RANGE`.
- A device with no stored config answers `total_len = 0` and no bytes.
- Reading is independent of the transfer session and always reflects the
  **active** config: chunks concatenated `0..total_len` reproduce the
  committed blob byte-for-byte (export ‚Üí import ‚Üí export is byte-stable).

## Keymap editing (v1.2)

Batched keymap read/write by (layer, grid position), gated by feature bit 7.
Phase 6 of docs/implementation-plan.md: keymap editing over both our
transports (USB and BLE) instead of depending on Vial.

### Keymap key space

- Distinct from the LED key space: keymap positions cover the full matrix
  grid, `key = row * keymap_cols + col`, `0 .. keymap_rows * keymap_cols`.
  The grid dimensions come from the capability extension (6 √ó 14 = 84 on
  the Glove80; layer count is the existing `layer_capacity`).
- On the Glove80, 80 of the 84 grid positions are physical keys; positions
  5, 8, 75 and 78 (rows 0 and 5, columns 5 and 8) are holes. Holes read
  back as `0x0000` (`KC_NO`) and accept writes like any other position
  (they are real matrix slots with no key behind them, exactly as Vial
  treats them); writing anything but `0x0000` to a hole is pointless but
  harmless.

### Action encoding

- Key actions travel as the **VIA/Vial 16-bit keycode encoding** (u16 LE) ‚Äî
  byte-for-byte the values Vial reads and writes over its own protocol, and
  converted in firmware by the exact same conversion Vial uses. Vial-made
  edits and host-protocol edits are therefore interchangeable: each sees
  the other's writes verbatim.
- Limits are inherited from that encoding: RMK `KeyAction`s with no VIA
  representation (some morse/tap-hold forms, unsupported keycode variants)
  read back as a best-effort mapping ‚Äî `0x0000` in the worst case ‚Äî exactly
  as Vial shows them. The write read-back (below) makes any lossy mapping
  visible immediately.
- Hosts can reuse existing QMK/Vial keycode tables; `0x0001` is `KC_TRNS`
  (transparent), `0x0000` is `KC_NO`.

### KEYMAP_READ (0x50)

Request payload (3 bytes):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `layer` |
| 1 | 1 | `start_key` ‚Äî first grid position to read |
| 2 | 1 | `max_count` ‚Äî max keycodes wanted in this response |

OK response payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `layer` echoed |
| 1 | 1 | `start_key` echoed |
| 2 | 1 | `count` |
| 3 | 2¬∑n | `count` √ó keycode (u16) at positions `start_key .. start_key + count` |

- `count = min(max_count, max_keymap_entries_per_op, positions left to the
  end of the grid)` ‚Äî chunked iteration: repeat with
  `start_key += count` until the grid (or the range you care about) is
  covered. `max_count = 0` is answered with `count = 0`.
- `layer ‚â• layer_capacity` or `start_key ‚â• keymap_rows * keymap_cols` ‚Üí
  `OUT_OF_RANGE`.
- Reads reflect the **live** keymap: Vial edits, host-protocol writes and
  compiled defaults all read back through the same runtime state.

### KEYMAP_WRITE (0x51)

Request payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `entry_count` |
| 1 | 4¬∑n | entries: `layer u8` + `key u8` + `keycode u16` |

OK response payload:

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | `entry_count` echoed |
| 1 | 2¬∑n | per-entry read-back: the keycode now stored at each written position, in request order |

- Validation is **all-or-nothing**: every entry's `layer`/`key` is checked
  first; any out-of-range entry rejects the whole batch with
  `OUT_OF_RANGE` and nothing is written. `entry_count` beyond
  `max_keymap_entries_per_op` ‚Üí `CAPACITY_EXCEEDED`.
- Entries may target any mix of layers. Later entries win on duplicate
  positions (writes apply in order).
- Applied writes go through the same update-and-persist path as Vial's
  keycode writes: the live keymap changes immediately (no reboot) and the
  per-key change is persisted in RMK storage.
- The read-back is the canonical encoding actually stored
  (decode ‚Üí apply ‚Üí re-encode). It differs from the request exactly when
  the requested keycode is not exactly representable ‚Äî hosts SHOULD compare
  and surface any difference.

## Frame layer (per-transport segmentation)

Every transport chunk (one HID report / one ATT write or notification):

| offset | size | field |
| --- | --- | --- |
| 0 | 1 | control: bits 0‚Äì6 = sequence (0-based, per message), bit 7 = FINAL |
| 1 | 1 | `chunk_payload_len` (bytes of message data in this chunk) |
| 2 | n | message bytes |
| 2+n | ‚Äî | padding (zeros) up to the transport's fixed chunk size, if any |

- Sequence restarts at 0 for each message; max 128 frames per message.
- One message in flight per direction per transport; interleaving is not
  allowed. A frame with sequence 0 always starts a new message (an
  incomplete previous message is dropped).
- A sequence gap or a chunk shorter than `2 + chunk_payload_len` aborts
  reassembly with an error.
- USB raw HID: chunk size 32, reports zero-padded (payload ‚â§ 30/report).
- BLE GATT: chunk size = negotiated ATT payload, assume ‚â• 20 (payload ‚â• 18
  per write). Frames need not be padded.
- The message header's `payload_len` cross-checks reassembly: total
  reassembled length must equal header length + `payload_len`.

## Transports

How the frame layer maps onto the physical transports the Glove80 RMK
firmware exposes (Phase 2). Both transports terminate on the **central (left)
half**; the peripheral has no host-facing transport. One request/response
exchange is in flight per transport at a time (enforced device-side: the next
request frame is not consumed until the previous response is fully framed
out).

### USB raw HID

- A dedicated vendor raw-HID interface on the keyboard's composite USB
  device (VID `0x16C0`, PID `0x27DB`), separate from the keyboard, composite
  and Vial interfaces.
- Match it by HID usage page **`0xFF88`**, usage **`0x01`** (Vial's raw-HID
  interface is usage page `0xFF60`/usage `0x61` ‚Äî do not confuse them; the
  opcode spaces collide).
- Reports: 32-byte IN and 32-byte OUT, no report IDs. Every report is one
  frame-layer chunk, zero-padded to 32 bytes (‚â§ 30 payload bytes/report).

### BLE GATT

- A custom primary service (deliberately **not** a HID service, so Web
  Bluetooth can reach it; it is not in the advertising payload ‚Äî Web
  Bluetooth clients must list it in `optionalServices`):

| role | UUID |
| --- | --- |
| service | `fc550001-f8e0-459f-b421-c254fc42b138` |
| request characteristic (host ‚Üí keyboard) | `fc550002-f8e0-459f-b421-c254fc42b138` |
| response characteristic (keyboard ‚Üí host) | `fc550003-f8e0-459f-b421-c254fc42b138` |

- Requests: **write-without-response** to the request characteristic, one
  frame-layer chunk per write, unpadded. Chunk size ‚â§
  `min(negotiated ATT payload, 257)` (257 = 2-byte frame header + the
  255-byte max chunk payload).
- Responses: **notifications** on the response characteristic (subscribe via
  CCCD first), one chunk per notification, unpadded, sized to
  `min(negotiated ATT payload, 257)` as observed when the request arrived.
- Like the rest of the GATT database, the service requires an encrypted
  (bonded) connection.

### Phase 2 device status

- Key indices `0..40` (left half) apply immediately on the central. Indices
  `40..80` (right half) are stored authoritatively on the central and
  forwarded to the peripheral over the split link (Phase 3): overlay writes
  answer `OK` when the peripheral is connected and the forward was
  dispatched, and `PARTIAL_APPLY` (listing the right-half keys) only while
  the peripheral is genuinely unavailable ‚Äî the stored cells then apply via
  the reconnect resync. `READ_OVERLAY` reports all 80 keys (TTLs included)
  from the central's stores. `CLEAR_OVERLAY` / `REPLACE_OVERLAY` with the
  peripheral offline answer `PARTIAL_APPLY` (for a bare right-half clear,
  with `pending_count = 0` per the overlay-ack rules above).
- `ENTER_BOOTLOADER`: target 0 (central) works via the Adafruit bootloader.
  Target 1 (peripheral) is forwarded over the split link (magic-guarded on
  that hop too): `OK` means the request was dispatched to a connected
  peripheral, which then reboots into its UF2 bootloader; `BUSY` means the
  peripheral is offline (or the split queue was momentarily full) and
  nothing happened ‚Äî retry.
